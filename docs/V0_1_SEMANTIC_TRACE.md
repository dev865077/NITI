# v0.1 Semantic Trace

This trace connects the mathematical cDLC claim to the deterministic Bitcoin
harness. It keeps formal proof, transaction evidence, and assumptions separate.

## Oracle Scalar To Bridge Signature

The core path is:

```text
oracle event and outcome
  -> oracle nonce point R_o
  -> outcome challenge e_x
  -> oracle scalar s_x
  -> attestation point S_x = s_x G
  -> adaptor point on the bridge
  -> completed bridge signature
```

In the deterministic harness this path is generated by:

```sh
npm run test:cdlc-smoke
```

Expected artifact kind:

```text
niti.v0_1_cdlc_smoke_transcript.v1
```

The transcript records the event id, selected outcome, oracle nonce point,
oracle public key, attestation point, attestation scalar, parent CET sighash,
bridge sighash, adaptor point, adaptor scalar, completed signature, extracted
secret, and wrong-outcome rejection.

Public evidence bundles commit to the serialized raw transactions and their
transaction identifiers; the verifier recomputes the transaction and signature
checks from those artifacts.

## Encoding Assumptions

Scalars are encoded as 32-byte hex strings reduced modulo the secp256k1 group
order. Curve points are SEC1 compressed points unless a field name explicitly
says `XOnly`. Taproot signatures are BIP340 Schnorr signatures over 32-byte
Taproot sighashes. Transaction identifiers are non-witness transaction ids for
SegWit/Taproot transactions; witness data changes signature validity but not
the transaction id committed by prepared descendants.

## Parent-Bridge-Child State Map

| State | Trigger | Evidence |
| --- | --- | --- |
| Parent prepared | Deterministic parent funding and CET templates are created. | `npm run test:cdlc-smoke` |
| Parent funded | Parent funding fixture exists and its signature verifies. | `niti.v0_1_parent_funding_fixture.v1` |
| Parent resolved | Oracle publishes the selected outcome scalar. | `oracle.activatingSignatureVerifies = true` |
| Parent CET completed | Selected scalar completes the parent adaptor signature. | `activationPath.parentCet.completedSignatureHex` and `oracle.activatingSignatureVerifies = true` |
| Bridge completable | Same selected scalar completes the bridge adaptor signature. | `activationPath.bridge.completion.completedSignatureVerifies = true` |
| Bridge confirmed or simulated | Bridge spends the parent edge output and creates child funding. | `chainSimulation.blocks` and bridge confirmation fields |
| Child funded | Child funding output exists as the bridge output. | `activationPath.bridge.output` equals `activationPath.childPreparedCet.input` |
| Child prepared | Child CET and refund spend the child funding output. | `child.preparedSpendChecks` |
| Timeout fallback | Bridge or child refund path matures after its locktime. | `edgeTimeoutRefund.timelockCheck` and adversarial matrix timeout checks |

Invalid transitions are rejected in Layer 3:

```sh
npm run test:bilateral-state-machine
npm run test:bilateral-wrong-path-replay
npm run test:adversarial-failure-matrix
```

## Negative Path Coverage

| Invalid input | Expected behavior | Evidence |
| --- | --- | --- |
| Wrong outcome scalar | Bridge completion fails. | `npm run test:cdlc-smoke` and `npm run test:adversarial-failure-matrix` |
| Random scalar | Bridge completion fails. | `npm run test:adversarial-failure-matrix` |
| Different event scalar | Bridge completion fails. | `npm run test:adversarial-failure-matrix` |
| Different oracle key scalar | Bridge completion fails. | `npm run test:adversarial-failure-matrix` |
| Completed signature replayed to another bridge sighash | BIP340 verification rejects it. | `npm run test:adversarial-failure-matrix` |
| Stale or reordered bilateral transcript | Setup validation rejects it. | `npm run test:bilateral-wrong-path-replay` |
| Missing retained state | Settlement aborts safely. | `npm run test:bilateral-state-retention` |
| Malformed counterparty adaptor data | Adaptor exchange rejects it. | `npm run test:bilateral-malformed-counterparty` |
| Missing lazy prepared-edge package | Holder activation aborts safely. | `npm run test:bilateral-lazy-activation` |

## Proof-To-Engineering Boundary

| Claim category | v0.1 status |
| --- | --- |
| Algebraic identity `S_x = s_x G` | Proved in SPARK/Ada finite models. |
| Adaptor verification and completion | Proved in SPARK/Ada finite models and demonstrated in TypeScript harnesses. |
| Wrong scalar non-completion | Proved in SPARK/Ada finite models and demonstrated in deterministic negative tests. |
| BIP340 verification implementation | Exercised by the TypeScript harness; not formally proven. |
| secp256k1 group law | Assumed through the cryptographic library and Bitcoin consensus model. |
| SHA-256 and tagged hashes | Assumed cryptographic primitives. |
| Transaction serialization and sighash binding | Exercised by deterministic transaction harnesses; not formally proven. |
| Bitcoin Core policy and public mempool relay | Demonstrated only where committed public or regtest evidence exists. |
| Wallet key management and backups | Out of scope for v0.1. |
| Oracle source integrity and uptime | Assumed outside the scalar relation. |
| Economic solvency | Separate product model work, not a v0.1 protocol proof. |

## Audit Rule

A statement about v0.1 is acceptable only if it says whether the behavior is
proved, demonstrated, modeled, assumed, or out of scope.
